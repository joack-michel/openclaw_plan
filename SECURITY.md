# 安全策略

用途：说明公开仓库中的安全边界、问题反馈要求和秘密信息处理方式。

## 公开仓库边界

请勿在 Issue、Pull Request 或公开提交中提供：

- 真实 Token、Cookie、密码或密钥；
- 生产数据库、运行日志、备份和会话；
- 个人记忆、Telegram ID、账号标识；
- 私有业务接口参数、设备信息或生产固定执行路径；
- 生产 Cron、Grant、Operation、Run 或 Session ID。

生产凭据应保存在仓库外的私有覆盖层或专用密钥管理器中。报告问题时只描述信息类别、脱敏后的错误、受影响文件和复现步骤，不要复制秘密原文。

## 风险边界

本模板采用四级风险模型：

- L0：只读、查询和诊断，直接执行；
- L1：已登记、低风险、作用域固定的能力，直接执行；
- L2：有影响但可恢复，或扩大权限的动作，确认一次；
- L3：资金、秘密、破坏性或不可逆动作，严格确认或拒绝。

`allowlist miss` 和未登记 executable 属于 `CONFIG_ERROR`，不能通过反复确认解决。此类失败不应创建 `WAIT_CONFIRM`，也不能写入成功去重状态。

不得为了减少确认而全局放行 Bash、Shell、Node、Python、任意脚本、任意第三方消息、支付、下单或凭据读取。

## 漏洞报告

可以提交：

- 脱敏后的错误类型和日志片段；
- 受影响版本和文件；
- 可复现步骤；
- 预期结果与实际结果；
- 不包含真实身份和秘密的最小示例。

发现可能导致资金损失、秘密泄露、权限扩大或任意代码执行的问题时，请避免在公开 Issue 中附带可直接利用的生产参数。
