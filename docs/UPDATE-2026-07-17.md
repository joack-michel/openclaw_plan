# 2026-07-17 更新说明

本次更新集中解决两个长期问题：

1. 安全模式过严，普通读取、REM 和已有自动化容易被误拦截；
2. 每新增一个执行型 Skill，都要手工修改多层白名单，维护成本高且容易漏配。

本说明只记录脱敏后的通用架构和验收结果，不包含生产路径、真实身份、Grant ID、Token、Cookie、日志、私有业务实现或外部平台账号。

## 一、安全模式调整

安全策略改为四级风险模型：

| 等级 | 类型 | 默认行为 |
|---|---|---|
| L0 | 只读、查询、诊断 | 直接执行 |
| L1 | 已登记、低风险、可恢复能力 | 直接执行 |
| L2 | 有影响但可恢复，或扩大权限 | 确认一次 |
| L3 | 资金、秘密、破坏性或不可逆操作 | 严格确认或拒绝 |

普通 Workspace、Skill、Memory、日志、Cron 和 Grant 查询不再因为局部策略异常而整体锁死。

完整性哈希异常继续采用：

```text
WARN_ONLY
```

即记录审计和警告，但不阻断基础诊断。

## 二、CONFIG_ERROR 与确认流程分离

此前 `allowlist miss` 可能被错误转换成确认请求，用户即使反复回复“确认”，主机侧仍然不会执行。

现在执行流程改为：

```text
Gate 风险判断
→ Host approvals 预检
→ 命中：继续执行
→ 未命中：CONFIG_ERROR
```

`CONFIG_ERROR` 会提供脱敏后的 capability、executable、argv 形状和拒绝层，但不会：

- 创建 `WAIT_CONFIRM`；
- 要求用户重复确认；
- 写入成功去重状态；
- 被误判为业务成功或失败。

未知 Shell、解释器、脚本和未登记 executable 仍然被拒绝。这是有意保留的安全边界。

## 三、本地恢复 CLI

新增或补全本地恢复能力：

```text
status
diagnose
approvals list
approvals add-fixed
approvals remove
policy validate
policy restore-backup
gateway restart
undo
```

恢复 CLI：

- 仅允许服务器本地受信任用户调用；
- 不监听网络；
- 不能由 Telegram 触发；
- 默认 dry-run；
- 修改前自动备份；
- 所有操作写入审计；
- 支持撤销最近一次变更；
- 拒绝通配符、Shell 控制符和宽泛解释器入口。

## 四、事务式 Skill 注册器

本次新增事务式 Skill 注册系统，解决每个 Skill 都要手工同步多层白名单的问题。

新的权限来源：

```text
execution-manifest.json
→ Skill Registry
→ Execution Gate
→ Host approvals
→ Gateway 运行时验证
```

注册器提供：

```text
skill inspect
skill register
skill update
skill verify
skill remove
skill list
```

### 核心特性

- 默认 dry-run；
- 原子事务；
- 自动备份和失败回滚；
- 重复注册幂等；
- Manifest 严格 schema 校验；
- executable、argv、cwd、超时和环境精确匹配；
- 更新时自动移除旧 approval；
- Gateway 重启后重新验证；
- 未注册入口继续返回 `CONFIG_ERROR`。

Gate 已改为数据驱动。普通业务 Skill 不再需要每次手工修改 capability resolver 或 risk resolver 源码。

## 五、现有 Skill 迁移

已对现有固定业务能力、REM 记忆扫描和福利任务编排进行注册表迁移与核验。

公开仓库只记录通用注册机制，不包含任何私有业务模块、生产 Manifest、实际 executable、真实 argv 或身份作用域。

迁移和验证过程没有触发下单或支付。

## 六、REM 真实 E2E

REM 已通过真实 Gateway 端到端验证：

```text
仅一次 exec
成功发现 7 个文件
未执行写入
未触发外部业务动作
```

此前另一轮完整扫描也验证了普通 Memory 文件可正常读取，缺失目录按 `SKIPPED` 处理，不再因为固定 `find` 入口出现 `allowlist miss`。

## 七、测试结果

事务式 Skill 注册完成后：

```text
102 passed, 0 failed
```

覆盖内容包括：

- Manifest 校验；
- dry-run 不写入；
- 注册、更新、删除事务；
- 幂等注册；
- 中途失败回滚；
- Registry、Gate 和 approvals 漂移检测；
- executable 与 argv 精确匹配；
- 未登记入口返回 `CONFIG_ERROR`；
- Gateway 重启后持续生效；
- 高风险保护未退化。

## 八、Git 与版本管理

本地 Execution Gate 已纳入 Git 管理。

事务式 Skill 注册实现对应本地提交：

```text
922465f feat: add transactional skill registration
```

公开仓库只同步脱敏通用代码、文档和模板，不上传生产 Registry、真实 approvals、备份、数据库、运行日志、身份范围或私有业务模块。

## 九、以后新增 Skill 的流程

以后新增执行型 Skill，只需要：

```text
1. 编写 execution-manifest.json
2. skill inspect
3. skill register（默认 dry-run）
4. skill register --apply
5. skill verify
```

详细说明见：

```text
docs/SKILL-REGISTRATION.md
```

## 十、仍待验证的项目

- 合法 Grant 周期和额度内的外部福利任务；
- 不包含生产秘密的完整恢复演练。

不得为了完成验收而重置 Grant、扩大授权、下单或支付。

## 总结

这次更新没有取消安全控制，而是把安全限制集中到真正危险的能力上，同时降低普通读取、REM、已有自动化和新增 Skill 的维护成本。

最终结构为：

```text
普通读取可用
固定能力可登记
权限扩大可审计
配置错误可诊断
失败可以回滚
支付、下单、秘密和破坏性操作继续严格保护
```
