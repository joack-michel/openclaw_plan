# 安全模型

本模板采用“低摩擦、可诊断、可恢复”的风险分级方式。

目标不是让所有操作都经过审批，而是把保护集中在真正高风险的能力上：模型误操作、重复执行、权限扩大、支付、下单、秘密泄露和破坏性操作。

## 四级风险

| 等级 | 典型能力 | 默认决策 |
|---|---|---|
| L0 | 只读、查询、诊断 | `ALLOW` |
| L1 | 已登记、低风险、可恢复的业务动作 | `ALLOW` |
| L2 | 有影响但可恢复，或会扩大权限的动作 | `CONFIRM` |
| L3 | 资金、秘密、破坏性或不可逆动作 | `DENY` 或严格确认 |

### L0：直接执行

包括普通 Workspace、Skill、Memory、日志、Cron、Grant、服务状态和 Git 状态的读取。

L0 不创建确认操作，不进入 `WAIT_CONFIRM`。完整性哈希不一致时记录 `WARN_ONLY`，不得阻断基础诊断。

类似凭据的路径、其他 Agent 私有目录和符号链接逃逸不能继承普通 Workspace 权限。

### L1：固定能力直通

包括已有 ACTIVE Grant 范围内的自动化、本人通知、REM 扫描、固定优惠权益任务和通过事务注册器登记的低风险业务 Skill。

L1 必须绑定结构化身份：

```text
channel
actor_id
agent_id
capability
executable
argv
resource scope
grant scope
```

固定业务入口必须精确匹配 executable 和参数。不得因为放行一个业务脚本而放行任意 Bash、Node、Python 或未知脚本。

L1 仍应保留并发锁、短时去重、固定作用域和审计。只有能够明确证明失败时才允许有限重试；结果不确定时禁止重试。

### L2：确认一次

包括修改核心配置、扩大 Grant、新增固定入口、修改自动化、删除普通文件和其他有影响但可恢复的操作。

确认应绑定 capability、目标、参数、作用范围和有效期，而不是只绑定容易因空格或命令形式变化而失效的完整 Shell 字符串。

### L3：严格保护

支付、下单、转账、秘密导出、开放公网、修改身份白名单、批量删除、系统自毁和绕过安全系统等能力继续严格保护。

明显破坏性行为直接拒绝。高风险但存在合理用途的操作必须经过明确、范围受限的确认。

## CONFIG_ERROR 与确认分离

`allowlist miss` 属于执行配置问题，不是用户尚未授权。

当 Gate 已批准，但主机侧 approvals 不允许 executable 或参数时，应返回：

```text
CONFIG_ERROR
```

错误信息应包含脱敏后的 capability、executable、argv 形状和拒绝层。

此类失败不得：

- 创建 `WAIT_CONFIRM`；
- 要求用户反复回复“确认”；
- 写入成功去重状态；
- 被误报为业务失败或成功。

未知 Shell、解释器、脚本和未登记 executable 默认也应返回 `CONFIG_ERROR`。日常文件操作优先使用原生 `read`、`write`、`edit` 和 `apply_patch`，而不是依赖任意 Shell。

## Host approvals 预检

L0/L1 exec 在真正执行前应进行主机侧 approvals 预检。

```text
Gate 风险判断
→ Host approvals 预检
→ 命中：执行
→ 未命中：CONFIG_ERROR
```

这样可以避免“Gate 已允许，但底层执行仍被拒绝”的确认死循环。

## Automation Grant

已有且有效的低风险自动化可在原授权范围内直接运行。

额度耗尽、周期未刷新或作用域不匹配时，应拒绝当前运行，不得自动新建、扩容、重置或绕过 Grant。

## 完整性监控

策略文件哈希不一致默认采用：

```text
WARN_ONLY
```

系统继续允许 L0 诊断和已登记的 L1 能力，同时记录审计信息。完整性监控用于发现异常，不应制造全局权限自锁。

## 本地恢复边界

恢复工具只能由服务器本地受信任用户调用，不监听网络，不能通过 Telegram 启用。

恢复工具可以提供状态、诊断、固定 approvals 管理、策略校验、备份恢复、Gateway 重启和撤销能力，但不得提供支付、下单、危险删除或秘密导出能力。

## 默认原则

```text
读取默认允许
已有固定能力默认允许
扩大权限需要确认
资金、秘密和破坏性操作严格保护
任何策略异常都必须可诊断、可恢复
```
